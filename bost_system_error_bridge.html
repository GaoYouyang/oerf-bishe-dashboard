BOST 真实数据质量与系统误差桥梁

这页把 NeRIF / PIV-BOST / 4D BOST 之外的一个关键问题单独拎出来：如果何远哲师兄给一小份真实 BOST 或 PIV-BOST 数据，你不能只问“能不能训练网络”，而要先判断 view geometry、mask/ROI、spatial resolution、calibration、位移场和重投影误差是否可信。

总判断

给师兄看的说法：我可以把真实 BOST 数据接入拆成一个数据质量控制层：先检查每个视角的几何、mask、位移场、背景纹理、空间分辨率和重投影误差，再决定是否跑 NeRIF、传统 baseline、PIV-BOST 补偿或 4D 低秩重构。

这条线非常适合本科毕业设计的“稳态贡献”：即使暂时拿不到完整内部代码，也能帮组里把数据接口和误差报告做清楚。它不替代何远哲主线，而是让主线更容易落到真实实验数据上。

真实数据先查什么

层级 要检查的字段 可能造成的问题 可做成的图或表
view geometry 相机外参、内参、视线方向、光路是否 pinhole / telecentric / endoscopic、自定义标定。 重建坐标系错位；少视角网络看似收敛但物理位置偏。 视角方向球面图、camera-to-volume 示意图、有效视域交集。
mask / ROI 火焰/喷流区域、背景有效区、遮挡、窗口边界、坏视角。 网络把无效背景当信号；传统 baseline 被边界伪影主导。 每视角 mask 覆盖率、坏点比例、ROI 一致性热图。
displacement field 位移算法、相关窗口/光流参数、confidence、outlier mask、单位。 位移噪声直接进入折射率梯度；PIV-BOST 补偿误差被放大。 位移幅值图、confidence 图、bad-vector ratio、reprojection residual。
spatial resolution 视角数、背景纹理尺度、光学系统放大率、景深、重构网格。 高频结构不是被算法“重构差”，而是系统本身不可分辨。 resolution phantom、edge/line-spread 曲线、view-count vs resolution 表。
calibration / forward model 光线模型、折射路径近似、Gladstone-Dale 常数、密度/温度转换假设。 物理量单位和尺度不可信；真实数据无法和论文指标对齐。 重投影误差分布、单位转换链、参数敏感性 tornado 图。
permission 能否写论文、能否公开图、能否给师兄外的人看、是否需要匿名化。 做了结果但不能用于毕业论文或组会材料。 manifest 中的 public / internal / confidential 标记。

本科可交付版本

M3C-1 数据 manifest 校验

把真实或公开 BOST 数据整理成统一 manifest：图像路径、相机参数、mask、位移场、单位、权限、可公开字段。缺字段时不崩溃，而是生成 checklist。

M3C-2 view health report

每个视角输出背景纹理、位移幅值、bad-vector ratio、mask 覆盖率和重投影残差。师兄能一眼看到哪个 view 可信、哪个 view 应该剔除。

M3C-3 resolution / forward-model sanity check

用 phantom 或 synthetic field 扫 view count、noise、grid size 和光线模型，报告空间分辨率、重构误差和高频结构可信边界。

问何远哲师兄的问题

  1. 组内真实 BOST/PIV-BOST 数据的相机模型更接近 pinhole、telecentric、endoscopic，还是自定义标定？
  2. 已有 displacement field 是互相关、光流、DaVis/PIVlab/OpenPIV，还是组内自写算法？是否有 confidence 或 outlier mask？
  3. 真实数据里最大误差通常来自位移提取、view registration、mask/ROI、光学窗口、空间分辨率，还是物性转换？
  4. 师兄更需要我做算法复现，还是先做数据 manifest、view health report 和误差图？
  5. 论文里能公开哪些真实数据图：原始背景、扰动图、位移场、mask、切片、指标，还是只能画示意图？
  6. 如果做 PIV-BOST，补偿发生在 raw image、displacement/correlation、calibration/disparity，还是 velocity-vector layer？
  7. 如果做 4D BOST，是否已有 frame-wise quality flag、bad frame、时序同步误差或 rank/noise/frame-count 的评估需求？

写作边界

不要写成：我解决了真实 OERF BOST 系统误差。

应该写成：我建立了一个面向 BOST/NeRIF 数据接入的系统误差检查与质量报告框架，用于在算法重构前量化 view geometry、mask、位移场、空间分辨率和重投影一致性。

这个边界能保护你的毕业设计：贡献不是“空泛 AI”，也不是“完整复现实验系统”，而是一个真实课题组可能会用到的前处理和质量控制工具。
