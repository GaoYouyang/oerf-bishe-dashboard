当前判断

公开站已经超过 1 GB 量级，GitHub Pages 仍能部署，但每次 artifact 上传和 Git 传输都会变慢。后续优先增加 DOI、出版社页、arXiv、机构库和阅读建议；只有“小型、明确开放、对主线明显有用”的 PDF 才继续缓存。最新 2.6 MB 的 OERF 4D-LII Figshare supplement 属于 P0 small：CC BY 4.0、与蔡组最新 4D diagnostics 直接相关，因此保留；NRIP 的 2.57 GB 无 LICENSE 代码仓库只放链接和结构审计，不镜像。

缓存决策规则

可以缓存

  • arXiv、NASA/NTRS、OSTI/DOE、HAL、Zenodo/Figshare/DataCite 明确开放文件。
  • 出版社页面或 Crossref 明确标 CC BY / CC BY 4.0，且下载端点返回真实 PDF。
  • 与 He/NeRIF/PIV-BOST/4D BOST 主线强相关，或能支撑本科 demo 的核心 baseline。
  • 文件体积通常小于 15 MB；超过 15 MB 必须有很强主线理由。

不要缓存

  • 学校 WebVPN、图书馆订阅、Springer/ACM/Elsevier/AIP 受控 PDF。
  • Crossref 只给 TDM license，或下载端点返回验证页、HTML 壳、403、CAPTCHA。
  • 文件超过 30 MB 且只是旁支背景，优先保留 DOI/官方入口。
  • 许可不清的 GitHub 代码、数据集压缩包、补充材料全量包。

体积分级

等级阈值处理方式例子
P0 small 0-5 MB 明确开放且相关即可缓存；生成封面和术语页。 arXiv 小论文、EPJ/MDPI 小 PDF、方法说明型论文。
P1 medium 5-15 MB 只缓存主线或高价值 benchmark；旁支只放入口。 NeRIF、Open BOS 方法邻居、关键光流/PIV baseline。
P2 large 15-30 MB 需要写明为什么进公开站；优先考虑只放 DOI/机构入口。 大型 NASA 报告、长篇综述、4D 表示法长论文。
P3 huge 30 MB 以上 原则上不再新增公开缓存，除非它是不可替代主线资料。 AOPIV-Net 67 MB、TBOS supersonic jets 62 MB、NASA BOS review 36 MB。

推荐发布结构

  1. 公开 GitHub Pages：HTML、图、demo 结果、小型开放 PDF、术语索引、阅读卡。
  2. 本地私有库：WebVPN/订阅 PDF、许可不清的全文、未确认下载、组内数据。
  3. 外部跳转：出版社 DOI、arXiv、机构库、DataCite/Zenodo/Figshare、官方代码仓库。
  4. 后续如果公开包继续逼近部署上限，改成 GitHub Actions 发布过滤包，而不是把完整工作仓库直接作为 Pages artifact。
